# アーキテクチャ vs 実装 不整合リスト

作成日: 2026-02-22

## ISSUE-001: スケジュール時間ウィンドウのデフォルト値が 12h → 24h に変更されている

**深刻度:** 中

**対象ファイル:** `config.yaml`（9〜10行目）、`src/config.py`（103〜104行目）

**アーキテクチャの仕様:**

```yaml
schedule:
  interval_hours: 12
  time_window_hours: 12
```

**実装の実態:**

```yaml
schedule:
  interval_hours: 24
  time_window_hours: 24
```

`config.py` のデフォルト値も同様に 24 になっている:

```python
interval_hours=int(schedule.get("interval_hours", 24)),
time_window_hours=int(schedule.get("time_window_hours", 24)),
```

アーキテクチャは 12 時間サイクル（06:00・18:00 実行）を定義しているが、実装は 24 時間に変更されており、アーキテクチャ文書が更新されていない。

## ISSUE-002: メールフッターに記事件数の上限超過通知が表示されない

**深刻度:** 中

**対象ファイル:** `templates/email.html`、`src/html_builder.py`、`src/main.py`

**アーキテクチャの仕様:**

> 上限を超えた場合、フッターに通知を追加する。例: *"Showing the 200 most recent articles out of 250 found."*

**実装の実態:**

`main.py` では切り捨て発生時にログ出力のみ行い、`build_email_html` に総件数を渡していない。テンプレートにもプレースホルダーが存在しない。フッターは以下のみ:

```html
<p>Generated by RSS News Filter | {{ generation_time }}</p>
```

送信されるメールに通知が一切表示されない。

## ISSUE-003: `Article` クラスが `@dataclass` ではなく `NamedTuple` で実装されている

**深刻度:** 低

**対象ファイル:** `src/__init__.py`

**アーキテクチャの仕様:**

```python
@dataclass
class Article:
    title: str
    link: str
    published: datetime
    source: str
    category: str  # from OPML folder
```

**実装の実態:**

```python
class Article(NamedTuple):
    ...
```

`NamedTuple` はイミュータブルであり、後からフィールドを変更しようとすると `AttributeError` が発生する。アーキテクチャが指定している `@dataclass` とは型の性質が異なる。

## ISSUE-004: `--fetch-only` モードが誤って状態ファイルを更新している

**深刻度:** 中

**対象ファイル:** `src/main.py`（91〜94行目）

**アーキテクチャの仕様:**

> `state/last_run.json` はパイプライン全体が正常完了した場合のみ更新する。正常完了とは以下のいずれか:
> - メール送信が成功した場合
> - `--dry-run` で HTML ファイルが保存された場合

**実装の実態:**

```python
if fetch_only:
    logging.info("[Main] Fetch-only mode completed")
    save_last_run_timestamp(config.output.state_file, now_utc())  # ← 不正な更新
    return 0
```

`--fetch-only` はデバッグ用モードであり、メール送信も重複排除も行われない。ここで状態ファイルを更新すると、次回の本番実行でその間の記事が取りこぼされる原因となる。

## ISSUE-005: `rss_fetcher.py` が全記事に対して WARNING を過剰に出力している

**深刻度:** 低

**対象ファイル:** `src/rss_fetcher.py`（71〜75行目）

**アーキテクチャの仕様:**

> タイムゾーンなしの日付の場合、UTC と見なして WARNING をログ出力する

**実装の実態:**

```python
parsed = entry.get("published_parsed") or entry.get("updated_parsed")
if parsed is not None:
    logging.warning("[RSS Fetcher] Naive published_parsed assumed UTC: %s", source_name)
    return datetime(*parsed[:6], tzinfo=timezone.utc)
```

`feedparser` は仕様上、`published_parsed` を常にタイムゾーンなしの `time.struct_time` で返す。そのため、この WARNING は正常に取得できた全記事に対して発火し、1 回の実行で数百件の無意味な WARNING が出力される。アーキテクチャの意図は「日付文字列がタイムゾーン情報を持たない場合」への対処であり、feedparser の通常動作への対処ではない。`DEBUG` レベルへの変更、もしくは `published_parsed` パス限定での警告抑制が必要。

## ISSUE-006: "記事なし" メッセージが "24 hours" を決め打ちしている

**深刻度:** 低

**対象ファイル:** `templates/email.html`（75行目）

**アーキテクチャの仕様:**

> フィルタリング・重複排除後に記事が残らない場合、"No articles found" メッセージを表示する

**実装の実態:**

```html
<p>No articles found in the last 24 hours.</p>
```

時間ウィンドウは `config.yaml` の `schedule.time_window_hours` で設定可能だが、テンプレートが "24 hours" を直書きしている。`build_email_html` が `time_window_hours` をテンプレートに渡していないため、設定値と表示が一致しない。

## ISSUE-007: `build_email_html` の `max_articles` パラメータが使われていない（デッドコード）

**深刻度:** 低

**対象ファイル:** `src/html_builder.py`（29〜36行目）、`src/main.py`（119行目）

**実装の実態:**

```python
def build_email_html(
    articles,
    template_path="templates/email.html",
    max_articles=0,          # ← デッドパラメータ
):
    if max_articles > 0:
        articles = articles[:max_articles]
```

`main.py` では `build_email_html` 呼び出し前にすでに切り捨てを行っている:

```python
capped = deduped[: config.email.max_articles_per_email]
...
html = build_email_html(capped)   # max_articles は渡していない
```

`max_articles` パラメータは実際のパイプラインで一切使用されておらず、切り捨て責務の所在を曖昧にしている。
